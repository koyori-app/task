# Redis 信頼境界（セッション・WebAuthn チャレンジ）

## 概要

本バックエンドは **Redis をセッションストアと同等の機密性** で扱う。侵害された Redis は、有効なログインセッションの窃取と同等の影響を及ぼす。

## Redis に保存されるデータ

| キー接頭辞 | 内容 | TTL |
|-----------|------|-----|
| （axum-session） | ユーザーセッション ID・属性 | セッション設定に依存 |
| `webauthn:reg:{user_id}` | パスキー登録セッション状態 | 5 分 |
| `webauthn:reg:lock:{user_id}` | 登録フロー排他ロック | 30 秒 |
| `webauthn:auth:{challenge_id}` | パスキー認証セッション状態 | 5 分 |
| `webauthn:auth:disc:{challenge_id}` | Discoverable 認証状態 | 5 分 |

## リスク

1. **チャレンジの窃取**: 攻撃者が Redis から登録/認証状態を読み取れると、正当なクライアントの代わりに ceremony を完了できる可能性がある（`GETDEL` によりワンタイム消費だが、読み取り専用侵害ではリプレイの余地が残る）。
2. **セッション固定・乗っ取り**: セッションキーの漏洩はアカウント乗っ取りに直結する。
3. **登録ロックのバイパス**: ロック TTL 切れ後の並行登録で上限チェックの競合が理論上起こり得る（TTL 30 秒・通常フロー 5 秒未満で緩和）。

## 推奨セキュリティ対策

- Redis を **プライベートネットワーク** に配置し、インターネットから直接到達不可にする。
- **AUTH / TLS**（`rediss://`）を有効化する。
- 本番では **専用 Redis インスタンス** を使用し、他アプリと DB 番号を共有しない。
- バックアップ・レプリカへのアクセス権を最小化する。
- チャレンジは **サーバー側のみ** が読み書きし、クライアント API から Redis キーを推測できない設計を維持する（現状: ランダム `challenge_id` UUID）。
